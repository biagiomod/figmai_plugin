❯ ace-lambda is not readying template.yml file for all the environment variables. (e.g. S3 bucket name, JWT secret etc, cors origin allow list) Do we need to explicitly define environment variable in the infrastructure while creating it, or is there way it will re-load or read template.yml file?                                  

⏺ The Lambda function never reads template.yaml at runtime. The YAML file is purely a deployment descriptor — SAM/CloudFormation reads it once during sam deploy and bakes the values into the Lambda function's configuration in AWS. After that, the function reads them as standard process.env.* variables.

  Here's the full flow:

  template.yaml          sam deploy             AWS Lambda config        Runtime
  ─────────────    ───────────────────────►   ─────────────────────   ──────────────
  Parameters:             CloudFormation         Environment:            process.env
    S3BucketName   ──►   resolves params   ──►    S3_BUCKET: "..."  ──►  .S3_BUCKET
    JwtSecret             and writes env         JWT_SECRET: "..."       .JWT_SECRET
    CorsAllowOrigins       vars to the           CORS_ALLOW_ORIGINS:...  etc.
                           function config

  ---
  Your two options for supplying the values

  Option 1 — samconfig.toml (recommended)

  Create infra/ace-lambda/samconfig.toml with the concrete values for your environment. SAM reads this automatically on sam deploy:

  version = 0.1

  [default.deploy.parameters]
  stack_name = "ace-lambda-private"
  region = "us-east-2"
  capabilities = "CAPABILITY_IAM"
  resolve_s3 = true

  parameter_overrides = """
    S3BucketName=modstar-figmai-config \
    S3Prefix=figmai/ \
    S3Region=us-east-2 \
    ConfigApiToken=<your-token> \
    JwtSecret=<your-secret> \
    JwtExpiryHours=8 \
    CorsAllowOrigins=https://your-ace-domain.com \
    ServiceName=ace-lambda \
    ServiceVersion=0.1.0
  """

  Then deploy with just:
  sam build && sam deploy

  Option 2 — Pass overrides directly on the CLI

  sam build && sam deploy \
    --parameter-overrides \
      S3BucketName=modstar-figmai-config \
      ConfigApiToken=xxx \
      JwtSecret=xxx \
      CorsAllowOrigins=https://your-ace-domain.com

  ---
  Important: secrets should NOT go in samconfig.toml committed to git

  For ConfigApiToken and JwtSecret, the safe pattern is to leave them out of the toml file and either:
  - Pass them on the CLI at deploy time
  - Pull them from AWS SSM Parameter Store (requires a small template change)
  - Use a local .env-style file that is gitignored

  The existing infra/config-api/samconfig.toml (which is committed) shows the pattern already in use for this project — worth checking what it does for the token before deciding your approach for ace-lambda.

 Question: How does auth work in the private environment? Is there a single file that can be updated with private environement specific paths, endpoints, credentials, etc. that remains seperate from future updates to the ACE code?  The goal is once the ACE, APIs, and S3 are fully functional with this single private setup file any future 
 public changes to the ACE codebase will simply replace the old files, with zero setup effort since the single private setup files remains.                                                                                                                                                                                               

  ---
  The single private config file: config.js

  public/config.js is already designed for exactly this purpose. It is loaded before app.js via a separate <script> tag, so it is never overwritten when ACE code is updated. You set it once:

  // public/config.js — your private deployment file, never touched by ACE updates
  window.__ACE_CONFIG__ = {
    apiBase:  'https://your-lambda.execute-api.us-east-2.amazonaws.com/figma-admin',
    authMode: 'bearer'
  };

  app.js reads this at startup:
  const API_BASE      = window.__ACE_CONFIG__.apiBase  || ''
  const ACE_AUTH_MODE = window.__ACE_CONFIG__.authMode || 'cookie'

  Every API call then automatically becomes apiBase + '/api/model', etc., pointing at your Lambda. The /figma-admin prefix is absorbed into apiBase — app.js itself never needs to know about it.

  For the backend, infra/ace-lambda/samconfig.toml is the equivalent private file — holds bucket name, JWT secret, API token, CORS origins. Both files are gitignored and survive ACE code updates untouched.

  ---
  How auth flows end-to-end in the private environment

  1. Browser loads static ACE  →  config.js sets apiBase + authMode: 'bearer'

  2. init() checks sessionStorage for 'ace_bearer_token'
     └─ not found → shows login form (or bootstrap form on first run)

  3a. First run:
      POST /figma-admin/api/auth/bootstrap  { username, password }
      ← { success, user, token }   ← JWT returned here

  3b. Subsequent runs:
      POST /figma-admin/api/auth/login      { username, password }
      ← { success, user, token }   ← JWT returned here

  4. JWT stored in sessionStorage['ace_bearer_token']

  5. Every _apiFetch() automatically adds:
     Authorization: Bearer <jwt>

  6. Lambda verifies JWT on every protected route (authUtils.js)

  7. GET /figma-admin/api/auth/me  →  returns { user, role, expiresAt }

  How does authntication functionality work in ace-lambda?                                                                                                                                                                                                                                                               

  Read 3 files (ctrl+o to expand)

⏺ Here's how authentication works across all three files and the full request lifecycle:

  ---
  Two token types, one auth guard

  Every request passes through handler.js → isAuthorized() in authUtils.js. It accepts either of two token types sent as Authorization: Bearer <token>:

  Request arrives
        │
        ▼
  Extract Bearer token from header
        │
        ├─ matches CONFIG_API_TOKEN? ──► AUTHORIZED (service/script access)
        │
        └─ valid JWT signed with JWT_SECRET? ──► AUTHORIZED (user session)
                      │
                      └─ neither? ──► 401 Unauthorized

  The service token (CONFIG_API_TOKEN) is a static shared secret set in samconfig.toml — used by scripts, CI, or direct API calls. No expiry, no user identity.

  The user JWT is short-lived (default 8h), contains { sub, username, role, allowedTabs }, and is issued only after a successful login.

  ---
  Routes that bypass the auth guard entirely

  These six routes in handler.js skip isAuthorized() completely — no token needed:

  GET  /figma-admin/api/auth/me
  GET  /figma-admin/api/auth/bootstrap-allowed
  POST /figma-admin/api/auth/bootstrap
  POST /figma-admin/api/auth/bootstrap/create-first-account
  POST /figma-admin/api/auth/login
  POST /figma-admin/api/auth/logout

  ---
  First-run bootstrap flow

  1. GET  /auth/bootstrap-allowed
          reads admin/users.json from S3
          returns { allowed: true }  ← no active users exist yet

  2. POST /auth/bootstrap  { username, password }
          validates: password ≥ 8 chars
          bcrypt.hash(password, 10)
          writes { version:1, users:[...] } to S3 admin/users.json
          calls signToken(user)  ← issues JWT immediately
          returns { success, user, token }  ◄── caller is logged in instantly

  ---
  Subsequent login flow

  POST /auth/login  { username, password }
        │
        ├─ reads admin/users.json from S3
        ├─ finds user by username (case-insensitive), checks !disabled
        │
        ├─ user not found?
        │     bcrypt.hash('__dummy__')  ← timing protection (prevents user enumeration)
        │     return 401
        │
        ├─ bcrypt.compare(password, passwordHash)
        │     no match → 401
        │
        └─ match → signToken(user) → return { success, user, token }

  JWT payload stored inside the token:
  { "sub": "u_abc123", "username": "biagio", "role": "admin", "allowedTabs": [], "exp": 1234567890 }

  ---
  Auth/me — verifying who is logged in

  GET /auth/me   Authorization: Bearer <token>
        │
        ├─ token = CONFIG_API_TOKEN?
        │     return { user: { username:'service', role:'admin' }, tokenType:'service' }
        │
        ├─ jwt.verify(token, JWT_SECRET) succeeds?
        │     return { user: { id, username, role }, allowedTabs, tokenType:'user', expiresAt }
        │
        └─ invalid / expired?
              return 401

  ---
  Logout

  POST /auth/logout
        └─ returns { success: true }
              JWT is stateless — there is no server-side session to destroy.
              The client is responsible for discarding the token from sessionStorage.
              Token becomes harmless after JWT_EXPIRY_HOURS (default 8h).

  ---
  User data storage

  All user records live in S3 at {S3_PREFIX}admin/users.json:

  {
    "version": 1,
    "users": [
      {
        "id": "u_abc123_xyz",
        "username": "biagio",
        "passwordHash": "$2a$10$...",
        "role": "admin",
        "disabled": false,
        "createdAt": "2026-03-19T...",
        "updatedAt": "2026-03-19T..."
      }
    ]
  }

  passwordHash is never returned to the client — safeUser() in userService.js strips it before any response.
