## Phase 0: Prerequisites (minimal refactors so the editor has writable targets)

### 0.1 Add assistants manifest and generator

**Goal:** Assistant list and metadata (including promptTemplate) live in a single JSON file; build produces the same runtime behavior.

**Files to add:**

- **custom/assistants.manifest.json** (or src/assistants/assistants.manifest.json)  
  - Structure: `{ "assistants": [ { "id", "label", "intro", "welcomeMessage?", "hoverSummary?", "tag?", "iconId", "kind", "quickActions", "promptTemplate" } ] }`.  
  - Populate from current index.ts (one-time migration: extract each ASSISTANTS entry into manifest; promptTemplate = the base string passed to mergeKnowledgeBase, i.e. the inline prompt without the mergeKnowledgeBase/appendDesignSystemKnowledge wrapper).

**Files to add or extend:**

- **scripts/generate-assistants-from-manifest.ts** (new)  
  - Reads custom/assistants.manifest.json (or src/assistants/assistants.manifest.json).  
  - Outputs **src/assistants/assistants.generated.ts** (or a segment) that exports ASSISTANTS array where each item’s promptMarkdown = appendDesignSystemKnowledge(mergeKnowledgeBase(id, promptTemplate)).  
  - Key order and formatting deterministic.

**Files to change:**

- **src/assistants/index.ts**  
  - Replace inline ASSISTANTS array with: `export { ASSISTANTS } from './assistants.generated'` (or re-export from generated). Keep listAssistantsByMode, getAssistant, etc. here; change listAssistantsByMode to use **config** for simple/content-mvp (see 0.2).
- **package.json**  
  - Add script: `"generate-assistants": "tsx scripts/generate-assistants-from-manifest.ts"`.  
  - Add to prebuild: `npm run generate-assistants` (before or after generate-custom-overlay).

**Acceptance criteria:**

- Build succeeds; plugin loads; Select Assistant modal and all assistants behave as before.  
- Changing custom/assistants.manifest.json and running generate-assistants updates runtime list and prompts.

### 0.2 Move modal visibility/order into config

**Goal:** “Simple” and “content-mvp” assistant lists come from config, not hardcoded in index.ts.

**Files to change:**

- **custom/config.json**  
  - Add under **ui**: `"simpleModeIds": ["general", "content_table", "design_critique", "design_workshop"]`, `"contentMvpAssistantId": "content_table"`.
- **scripts/generate-custom-overlay.ts**  
  - Extend CustomConfig type and emitted customConfig to include ui.simpleModeIds and ui.contentMvpAssistantId (with safe defaults if missing).
- **src/custom/generated/customConfig.ts**  
  - Regenerated by running generate-custom-overlay; will include new ui fields.
- **src/assistants/index.ts**  
  - In listAssistantsByMode: for content-mvp, use customConfig?.ui?.contentMvpAssistantId ?? 'content_table'; for simple, use customConfig?.ui?.simpleModeIds ?? ['general', 'content_table', 'design_critique', 'design_workshop'] and filter/sort ASSISTANTS by that array.

**Acceptance criteria:**

- Changing simpleModeIds or contentMvpAssistantId in config.json and regenerating changes which assistants appear in simple/content-mvp mode.

### 0.3 Optional: config.dev.json or dev section for CONFIG overlay

**Defer** unless we want feature flags and dev scopes editable without code. If done later: add custom/config.dev.json (or a "dev" key in config.json) and a small generator that emits a TS overlay; CONFIG in config.ts merges that. Not required for Phase 1 editor.

## Phase 1: Admin Config Editor – server and model

### 1.1 Local Node server

**Goal:** Express (or similar) server that serves the UI and provides read/write API for the editable model.

**Files to add:**

- **admin-editor/server.js** or **admin-editor/server.ts** (or under scripts/admin-editor/)  
  - CWD = repo root (figmai_plugin).  
  - GET /api/model: read custom/config.json, custom/assistants.manifest.json, custom/knowledge/*.md, docs/content-models.md, custom/design-systems/*/registry.json; build AdminEditableModel; return JSON. Validate on load; include validation errors in response.  
  - POST /api/validate: body = AdminEditableModel; run Zod (or equivalent) validation; return { errors, warnings }.  
  - POST /api/save: body = AdminEditableModel; validate; if invalid return 400; else write custom/config.json, custom/assistants.manifest.json, custom/knowledge/<id>.md, docs/content-models.md, custom/design-systems/<id>/registry.json (deterministic, with backups); run npm run generate-assistants and npm run generate-custom-overlay (and optionally generate-presets); return { success, filesTouched, backupPath, summary }.  
  - Optional: GET /api/diff: return patch summary (e.g. list of files that would change) without writing.

**Files to add:**

- **admin-editor/package.json** (if separate package) with express, zod, and minimal deps. Or add deps to root package.json and run server from scripts.

**Acceptance criteria:**

- GET /api/model returns the full model and any load-time validation errors.  
- POST /api/validate returns errors/warnings for invalid payloads.  
- POST /api/save writes only the intended files, creates backups, runs generators, and returns a clear summary.

### 1.2 Schema (Zod) and deterministic write helpers

**Files to add:**

- **admin-editor/schema.ts** (or shared schema in a package the server and UI can use)  
  - Zod schemas for PluginConfig, AssistantDef, QuickActionDef, AdminEditableModel, ContentModelDef (if included).  
  - Field descriptions for UI hints.

- **admin-editor/write.ts** (or server/write.ts)  
  - Functions: writeConfig(config), writeAssistantsManifest(assistants), writeCustomKnowledge(id, body), writeContentModels(contentModels), writeRegistry(id, registry).  
  - All JSON: fixed key order, 2-space indent.  
  - Backups: copy file to custom/.backups/<filename>.<iso-timestamp> before write; prune to last N backups per file.

**Acceptance criteria:**

- Round-trip: load model → save model → load again; model is equivalent (and JSON/key order stable).  
- Backups are created and listed in save response.

## Phase 2: Admin Config Editor – UI

### 2.1 Static HTML + JS (or small framework)

**Goal:** Browser UI that loads the model, shows schema-driven forms, and calls /api/validate and /api/save.

**Files to add:**

- **admin-editor/public/index.html** (or admin-editor/index.html)  
  - Single-page app: fetch GET /api/model on load; render sections (Config, Assistants, Custom knowledge, Content models, Registries).  
  - Forms generated from schema (field types, descriptions).  
  - Buttons: “Validate only”, “Save” (with confirmation).  
  - After save: show “Files touched”, “Backup path”, “Next steps” (run generator if needed, build plugin, test in Figma, publish).

### 2.2 Schema-driven form rendering

**Goal:** Avoid hand-building every form; derive from Zod schema (or a JSON schema exported from Zod).

**Implementation options:**

- Use a library that renders forms from JSON Schema (e.g. react-jsonschema-form, or a lightweight custom renderer).  
- Or minimal: one section per top-level key (config, assistants, customKnowledge, contentModels, registries); each section has a known layout (e.g. config = nested key/value; assistants = list of cards with fields; customKnowledge = textarea per id).

**Acceptance criteria:**

- User can edit config, assistant list, custom KB, content models, and registries in the UI.  
- Validate-only shows errors/warnings without saving.  
- Save shows confirmation and then summary + next steps.

## Phase 3: Polish and documentation

### 3.1 “Next steps” and generator wiring

- After save, server runs generate-assistants and generate-custom-overlay (and optionally generate-presets).  
- UI always shows: “Ask the plugin owner to: 1) Run `npm run prebuild` (or generate-assistants + generate-custom-overlay) if not run automatically, 2) Build and test the plugin in Figma, 3) Publish if appropriate.”  
- Do **not** run full plugin build or publish from the editor.

### 3.2 Optional auth and audit

- If adding auth: custom/admin-accounts.json (user + passwordHash), login gate, session cookie; custom/admin-audit.log (timestamp, user, files touched) on save.  
- Document in README that the editor is for contributors with local access and that optional auth is for accountability only.

### 3.3 README and docs

- **admin-editor/README.md**: how to run the server, open the browser, use Validate and Save, and where backups live.  
- **docs/ADMIN_CONFIG_EDITOR_AUDIT.md**, **docs/ADMIN_CONFIG_EDITOR_ARCHITECTURE.md**, **docs/ADMIN_CONFIG_EDITOR_IMPLEMENTATION_PLAN.md**: already created; keep them updated as implementation proceeds.

**Acceptance criteria:**

- A new contributor can follow the README to run the editor and safely edit config and assistants.  
- Save response clearly states what was written and what the owner should do next.

## Acceptance criteria for “done”

1. **Phase 0:** Plugin builds and runs; assistant list and modal visibility/order are driven by config + manifest; no behavior change for end users.  
2. **Phase 1:** GET /api/model returns full model; POST /api/validate and POST /api/save work; backups created; generators run after save.  
3. **Phase 2:** User can edit all intended settings in the browser; Validate and Save work; summary and “next steps” are shown.  
4. **Phase 3:** README and docs are sufficient for a contributor to run and use the editor; optional auth documented.  
5. **Overall:** The editor does **not** compile or publish the plugin; it confirms successful updates, shows what changed, and instructs the user to ask the plugin owner to build/test/publish.

## Go/No-Go judgment

**Go.** The Admin Config Editor is feasible with minimal disruption provided:

1. **Refactor first (Phase 0):** Introduce custom/assistants.manifest.json and move modal visibility into config. This gives a single, file-based assistant source and avoids editing TypeScript from the tool.  
2. **Editor only writes to files the audit identified:** config.json, assistants.manifest.json, custom/knowledge/*.md, content-models.md, design-systems registry.json. Generated TS is produced by existing (or new) generators after save.  
3. **No plugin build or publish** from the editor; only file writes + generator runs + clear “next steps” for the owner.

---

## Top 3 risks

1. **Manifest migration and generator correctness**  
   Extracting the current ASSISTANTS array (including inline prompts and mergeKnowledgeBase/appendDesignSystemKnowledge usage) into a manifest and generating identical behavior is easy to get wrong. Mitigation: one-time migration script that outputs manifest from current index.ts; compare runtime ASSISTANTS (or promptMarkdown) before/after in tests or manual check.

2. **Round-trip and format stability**  
   content-models.md is Markdown; round-tripping from a parsed structure back to Markdown can change formatting or lose nuance. Mitigation: either keep a strict Markdown format in the editor (deterministic sections/order) or introduce content-models.json as the source and generate .md from it so the editor only touches JSON.

3. **Scope creep (CONFIG.dev, feature flags, selection policies)**  
   Exposing every knob (e.g. CONFIG.dev.debug.scopes) in the editor increases surface area and maintenance. Mitigation: limit Phase 1 to config.json, assistants manifest, custom knowledge, content models, and registries; document other settings (feature flags, dev, selection policy) in the audit and leave them as code unless there is a clear need to make them editable.
