How should Strike Teams own their code?

This is the most important architectural decision. Pick the structure that matches how your teams will actually work.

A

One repo, directory isolation Recommended

All Strike Teams work in the same GitHub repo. Each assistant lives in its own directory. GitHub CODEOWNERS enforces who can approve changes to each directory. The Core Code Team owns the shared layer.

figmai-plugin/
├── src/core/                 ← Core Code Team
├── src/sdk/                  ← Core Code Team
├── src/assistants/
│   ├── evergreens/          ← Evergreens Team
│   ├── analytics/           ← Analytics Team
│   ├── accessibility/       ← A11y Team
│   └── prompt-to-screen/   ← P2S Team
└── CODEOWNERS             ← ownership map

Pros

  • Minimal tooling complexity
  • Works great with VSCode + Copilot
  • One CI pipeline
  • Easiest migration path
  • Standard InnerSource pattern

Cons

  • Teams share a git history
  • PRs need Core Team review for cross-cutting changes
B

Monorepo with npm workspaces

Each assistant is a proper npm package inside a workspace. TypeScript project references enforce strict compile isolation. A failed assistant package never breaks the others.

figmai-plugin/
├── packages/
│   ├── core/                  @figmai/core
│   ├── assistant-sdk/         @figmai/sdk
│   ├── assistant-evergreens/  @figmai/evergreens
│   ├── assistant-analytics/   @figmai/analytics
│   └── assistant-p2s/         @figmai/p2s
└── package.json           # workspace root

Pros

  • Strictest compile isolation per assistant
  • Industry-standard monorepo pattern
  • Tooling ecosystem (Nx, Turborepo)

Cons

  • Significant tooling overhead upfront
  • More complex build pipeline to migrate
  • Steeper learning curve for new teams
C

Separate repos (federated)

Each assistant lives in its own GitHub repository. The core repo pulls them in via git submodules or a build-time fetch. Maximum ownership separation, maximum complexity.

github.com/org/figmai-core         Core repo
github.com/org/figmai-evergreens   Evergreens repo
github.com/org/figmai-analytics    Analytics repo
github.com/org/figmai-a11y         A11y repo

# Core clones all repos at build time
# or uses git submodules

Pros

  • True repo-level ownership
  • Fine-grained GitHub access control

Cons

  • Git submodules are error-prone
  • Separate CI per repo
  • Much harder to refactor shared types
  • Not recommended for VSCode + Copilot teams

Click your preference, then respond in the terminal. My recommendation is A — it maps directly to how your teams will work and is the easiest migration from the current codebase.
