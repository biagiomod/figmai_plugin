Three approaches to the extensibility architecture

All three use single-repo directory isolation (your choice A). They differ in how assistants plug into the core and how the compile gate works.

A

Directory SDK + per-assistant compile gate Recommended

Each assistant lives in its own directory and exports a single index.ts that the core reads at build time. A new build-assistants script validates each one independently — if an assistant's TypeScript fails, its previous generated file is kept and the build continues.

src/sdk/index.ts             ← public contract (Core owns)
  exports: AssistantDefinition, AssistantHandler,
           QuickAction, KnowledgeBase...

src/assistants/evergreens/
  index.ts                    ← exports { definition, handler }
  handler.ts                  ← Strike Team logic
  knowledge.md               ← KB content
  definition.ts              ← manifest entry (typed)

scripts/build-assistants.ts
  for each assistant/:
    try tsc → generate {name}.generated.ts ✓
    catch → keep old {name}.generated.ts, log error
→ core assembles all .generated.ts files

Pros

  • True per-assistant compile isolation
  • SDK is a typed contract — breaks loudly if violated
  • New teams get a clear template to copy
  • Preserves existing build-time model
  • Local-first: no CI needed

Cons

  • Requires building the SDK + build script (one-time Core Team work)
  • Teams need to learn the SDK interface
B

Directory isolation only — keep shared manifest

Move each assistant's handler + knowledge into its own directory, but keep contributing to the central custom/assistants.manifest.json. No new SDK, no per-assistant build gate. Simpler migration, weaker isolation.

src/assistants/evergreens/
  handler.ts                  ← handler (moved here)
  knowledge.md               ← KB (moved here)

custom/assistants.manifest.json
  ← still one big file, all teams edit here
  ← no per-assistant compile isolation

Build: unchanged — still one big tsc run
If Evergreens handler breaks → whole build fails

Pros

  • Very easy migration — just move files
  • No new tooling or SDK to build
  • Teams can start immediately

Cons

  • No compile isolation — one team's error breaks everyone
  • Manifest is a merge conflict hotspot
  • No typed contract for Strike Teams
C

Auto-discovery via assistant registry function

Each assistant directory exports a register() function. A discovery script scans src/assistants/*/index.ts automatically and calls register() to build the full registry. No manifest file at all — the directory is the manifest.

src/assistants/evergreens/index.ts
  export function register(sdk: FigmAISDK) {
    sdk.registerAssistant({ id, label, ... })
    sdk.registerHandler(handler)
  }

scripts/discover-assistants.ts
  glob src/assistants/*/index.ts
  → call register() on each
  → generate combined registry

Adding assistant = adding a directory. Done.

Pros

  • Frictionless: drop a folder in and it's registered
  • No manifest editing
  • Clean extension model (like VS Code)

Cons

  • Harder to debug — "magic" auto-loading
  • Discovery errors can be opaque
  • Diverges more from current architecture
  • More Core Team work to build correctly
Why A? It gives you per-assistant compile isolation (your stated requirement), a typed SDK contract that breaks loudly when a Strike Team violates the interface, and a clear copy-this template for new teams — all without breaking the existing build-time model. B is too fragile (one bad handler breaks everyone). C is elegant but requires more Core Team work and is harder to debug. A threads the needle.
