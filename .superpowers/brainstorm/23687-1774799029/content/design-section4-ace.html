Change 2 — User records gain an assistantScope field

CORE TEAM USER (admin — no scope = all)
{
  "username": "biagio",
  "role": "admin",
  "assistantScope": []  ← empty = all
}
STRIKE TEAM USER (scoped)
{
  "username": "alice",
  "role": "editor",
  "assistantScope": ["content_table"]
                      ← only Evergreens
}

Change 3 — ACE UI filters to show only scoped content

Config All Assistants All KBs Design Systems Users
assistantScope: [] → no filter applied
EVERGREENS TEAM — scoped view
Evergreens Assistant Evergreens KBs Evergreens Knowledge
assistantScope: ["content_table"] → filter applied at load + save
Config, Design Systems, Users tabs hidden

Where scoping is enforced (both UI and backend)

🔒
API load (GET /api/model)
Returns only the assistant entries + KBs within the user's assistantScope. Other assistants are stripped from the response.
🔒
API save (POST /api/save)
Save payload is validated against assistantScope. Any attempt to write to an out-of-scope assistant returns 403. Backend enforces — not just the UI.
🔒
KB routes (GET/POST /api/kb/...)
KB reads and writes are scoped to the assistant's own knowledge-bases/ directory. Cross-assistant KB access is blocked.
✓
ace-lambda (private deployment)
Same scoping logic applies to the Lambda-backed API. assistantScope is stored in users.json and checked on every protected route.
Onboarding a new Strike Team: Core Team admin creates their ACE user account, sets role: "editor" and assistantScope: ["their_assistant_id"]. That's it. They log in and see only their assistant. No other configuration needed.
