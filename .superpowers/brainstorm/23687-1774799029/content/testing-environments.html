How should Strike Teams test their changes in Figma?

Teams need to run a real Figma plugin build with their updates — without waiting on Core Team. Here are three complementary approaches.

A

Local dev build — zero infrastructure Always works

Figma natively supports loading a plugin from a local folder during development ("Development Plugins"). Any Strike Team member can clone the repo, build locally, and test immediately — no CI, no approval needed.

# 1. Clone and install (one-time)
git clone ... && npm install

# 2. Make changes in your directory
src/assistants/evergreens/ ← edit freely

# 3. Build (no S3 needed)
npm run build:offline

# 4. Load in Figma
Figma → Plugins → Development → Import from manifest
→ select build/manifest.json

✓ Full plugin. Your changes. Your Figma file.

Pros

  • Works today, no new infrastructure
  • Completely independent of Core Team
  • True end-to-end test in real Figma

Cons

  • Each team member needs repo cloned + Node.js
  • All assistants compile together (team sees all)
B

CI branch builds — shareable test artifacts Recommended add-on

GitHub Actions builds the plugin on every push to a feature branch. The build artifact (main.js + ui.html + manifest.json) is uploaded and linked in the PR. Anyone on the team downloads it, drops it in a folder, and loads it in Figma — no local build setup required.

Strike Team pushes to feature/evergreens-update
   ↓
GitHub Actions: npm run build:offline
   ↓
Uploads build/ as artifact on PR
   ↓
Team member downloads ZIP → loads in Figma
   ↓
Tests in Figma ✓ — no local Node.js setup needed

Pros

  • Non-technical team members can test
  • Shareable — whole team tests same build
  • Catches compile errors in CI automatically

Cons

  • Requires GitHub Actions setup (one-time, easy)
  • ~1-2 min wait per push
C

Separate Figma test plugin per team

Each Strike Team registers their own Figma plugin (separate plugin ID) for testing. They publish their own "dev edition" of the plugin independently. Production plugin is managed by Core Team only.

Production plugin: FigmAI (Core Team owns)
Evergreens test plugin: FigmAI-Evergreens-Dev
Analytics test plugin: FigmAI-Analytics-Dev

Each team publishes to their own plugin ID.
No coordination with Core Team for testing.

Pros

  • Completely isolated from production plugin
  • Teams can publish independently

Cons

  • Multiple Figma plugin registrations to manage
  • Test plugin drifts from production
  • Confusing for end users if they install wrong one
💡 These aren't mutually exclusive
A is always available and is the primary workflow for developers. B is a great add-on for non-developers on a Strike Team (designers, content strategists, etc.) who need to test without setting up a local environment. You can start with A and add B when teams grow.

You can select multiple. My recommendation: A + B together.
